## Lessons Learned (สิ่งที่ต้องจำ)

### Railway Deployment
- Next.js 16+ ต้องการ Node.js >= 20.9.0
- nixpacks.toml format:
  ```toml
  [phases.setup]
  nixPkgs = ["nodejs_20"]  # ไม่ต้องใส่ npm แยก

  [phases.build]
  cmds = ["npm run build"]

  [start]
  cmd = "npm run start"
  ```
- ถ้า build fail → ดู logs จาก Railway Dashboard → download JSON → อ่านหา error
- package.json ต้องมี: `"engines": { "node": ">=20.9.0" }`
- start script: `"start": "next start -p ${PORT:-3000}"`

### 🔐 Curl Login Skill (ทักษะบังคับ!)
**ใช้กับทุก service ที่ต้อง login - ห้ามใช้ browser automation ถ้าไม่จำเป็น**

**ขั้นตอน:**
1. 🔍 **หา API endpoint จาก JS bundle**
   ```bash
   curl -s "https://example.com/app/login" | grep -oE '/app/[^"]+\.js'
   curl -s "https://example.com/app/js/index.xxx.js" | grep -oE '"[^"]*login[^"]*"'
   ```
2. 🎯 **ลอง common endpoints**
   - `/accounts/login`
   - `/api/auth/login`
   - `/api/v1/auth/login`
   - `/api/login`
3. 📤 **ส่ง credentials (ใช้ไฟล์ถ้ามี special chars)**
   ```bash
   # สร้างไฟล์ JSON (หลีกเลี่ยง escape issues)
   echo '{"email":"x@y.com","password":"pass!@#"}' > /tmp/login.json
   curl -s -X POST "https://api.example.com/login" \
     -H "Content-Type: application/json" \
     -d @/tmp/login.json
   ```
4. 🔑 **รับ token และใช้งาน**
   ```bash
   # ส่วนใหญ่ใช้ Bearer token
   curl -s "https://api.example.com/me" \
     -H "Authorization: Bearer {token}"
   ```
5. 📝 **สำรวจ API endpoints จาก JS bundle**
   ```bash
   curl -s "https://example.com/app/js/index.xxx.js" | grep -oE '"/[a-z_]+/[a-z_]*"' | sort -u
   ```

**ห้ามลืม:** Login สำเร็จ → บันทึก API structure ลง identity.md ทันที!

**สิ่งสำคัญที่ต้องรู้:**

   - ไม่มี Gateway = ใช้ API ไม่ได้ = gen key auto ไม่ได้
   - Gateway เชื่อม Lock (Bluetooth) → WiFi → Cloud → API
   - ราคา Gateway ~$15-25

   - TTLock Gateway ใช้ได้เฉพาะ TTLock Lock
   - Tuya Gateway ใช้ได้เฉพาะ Tuya Lock
   - ห้ามซื้อคนละยี่ห้อ!

3. **TTLock API Endpoints:**
   - `/v3/keyboardPwd/add` - สร้างรหัสชั่วคราว
   - `/v3/keyboardPwd/get` - ดึงรหัส
   - `/v3/keyboardPwd/delete` - ลบรหัส
   - `/v3/lock/unlock` - ปลดล็อคระยะไกล
   - Docs: https://euopen.ttlock.com/doc/api/

4. **ราคา TTLock (Alibaba):**
   - Lock: $18-30/ตัว (~700-1,000 บาท)
   - Gateway: $15-25/ตัว

### 🔬 การศึกษา OpenClaw / Moltbook

**OpenClaw คืออะไร:**
- Open-source AI agent ที่รัน local
- 68,000+ GitHub stars
- Multi-channel: WhatsApp, Telegram, Discord, Slack, LINE
- 1,715+ community skills
- Heartbeat system (ตื่นเองทำงาน)
- Self-improving (เขียน skill ใหม่ให้ตัวเอง)

**Moltbook คืออะไร:**
- Social network สำหรับ AI agents
- 1.5 ล้าน agents ลงทะเบียน
- Agents post, comment, upvote กันเอง
- มนุษย์ได้แค่ดู
- "OAuth สำหรับ AI" - identity system

**ข้อจำกัดของ OpenClaw/Moltbook:**
- Security vulnerabilities (prompt injection, database leak)
- Karpathy: "don't recommend running on your computers"
- ไม่ custom สำหรับธุรกิจเฉพาะทาง

**สิ่งที่เราเรียนรู้และนำมาใช้:**
- Heartbeat/Proactive system
- Multi-agent architecture
- Shared memory concept
- Goal-based autonomous operation
- แต่ทำให้ปลอดภัยกว่า และ custom สำหรับธุรกิจที่พัก

**บทเรียน:**
```
❌ ผิด: Deploy แล้วรอ ไม่ดู logs
❌ ผิด: เดาว่าผิดตรงไหน แก้มััวๆ
❌ ผิด: ถาม user ว่าเกิดอะไรขึ้น

✅ ถูก: ไปดู View Logs ใน Railway Dashboard ทันที
✅ ถูก: อ่าน Build logs, Deploy logs, Runtime logs
✅ ถูก: หา error message ที่แท้จริง
✅ ถูก: แก้ตรงจุด แล้ว redeploy
```

### ขั้นตอน Debug Railway Deployment

```
1. Deploy ไป Railway (railway up)
          │
          ▼
2. ไปที่ Railway Dashboard
   https://railway.com/project/<project-id>
          │
          ▼
3. คลิก Service → Deployments tab
          │
          ▼
4. คลิก "View logs" ของ deployment ล่าสุด
          │
          ▼
5. ดูว่า fail ตรงไหน:
   - Build failed? → ดู build logs
   - Deploy failed? → ดู deploy logs  
   - Healthcheck failed? → ดู runtime logs
          │
          ▼
6. หา error message → แก้ code → redeploy
          │
          ▼
7. ทำซ้ำจนกว่าจะ success
```

### Logs ที่ต้องดู

| Log Type | ดูอะไร |
|----------|--------|
| **Build Logs** | npm install errors, syntax errors |
| **Deploy Logs** | Container startup, port binding |
| **Runtime Logs** | App crashes, exceptions |
| **Healthcheck** | ทำไม app ไม่ตอบ / ตอบช้า |

### CLI Commands ที่ใช้ได้

```bash
# ดู logs ผ่าน CLI
railway logs

# ดู build logs
railway logs --build

# ดู deployment logs
railway logs --deployment

# ดู variables
railway variables
```

### สิ่งที่ทำให้ Healthcheck Fail บ่อย

1. **PORT ไม่ถูก** — ต้องใช้ `process.env.PORT`
2. **App crash ก่อน listen** — syntax error, module not found
3. **Timeout** — App start ช้าเกิน 30 วินาที
4. **Wrong healthcheck path** — Default คือ `/`

### ตัวอย่างจาก Oracle Agent

**Error ที่เจอ:**
```
Attempt #1 failed with service unavailable
Healthcheck failed!
```

**สาเหตุจริง:**
- Template literal ใน SYSTEM_PROMPT มี escape characters ผิด
- App crash ตอน parse string

**วิธีแก้:**
- อ่าน logs → เห็นว่า app ไม่ start
- Test locally → หา error
- แก้ code → redeploy → success

---

**จำไว้: ไม่มี logs = ไม่มีทาง debug = ไม่มีวัน deploy สำเร็จ**
